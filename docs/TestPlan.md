<a name="introduction"/>

## 1. Введение

Данный документ описывает план тестирования для веб-приложения "Library Data Management". Документ предназначен для тестировщиков и разработчиков данного проекта. Цель тестирования — проверка соответствия реального поведения программы и ее ожидаемого поведения, которое заложено в ее функциональности и бизнес-логике.

<a name="items"/>

## 2. Объект тестирования

В качестве объектов тестирования выделяются следующие функциональные блоки:

*   Управление сущностью "Книги" (CRUD: создание, чтение, обновление, удаление).
*   Управление сущностью "Авторы" (CRUD).
*   Управление сущностью "Категории" (CRUD).
*   Навигация по приложению (переключение вкладок).
*   Функциональность поиска по каждой сущности.
*   Взаимодействие с серверной частью (API).

<a name="quality"/>

## 3. Атрибуты качества

*   **Функциональность:**
    *   **Функциональная полнота:** Приложение должно выполнять все заявленные CRUD-операции для всех трех сущностей (Книги, Авторы, Категории).
    *   **Функциональная корректность:** Приложение должно корректно отправлять запросы на API и отображать полученные данные. Операции должны приводить к ожидаемым изменениям данных.
*   **Надежность:**
    *   **Отказоустойчивость:** Приложение должно корректно обрабатывать ошибки от сервера (например, ошибки валидации, недоступность сервера) и информировать об этом пользователя, не прекращая свою работу.
*   **Удобство использования:**
    *   **Понятность:** Пользовательский интерфейс, построенный на Ant Design, должен быть интуитивно понятным.
    *   **Обратная связь:** Приложение должно предоставлять пользователю обратную связь о статусе операций (загрузка, успех, ошибка) с помощью индикаторов загрузки и всплывающих сообщений.

<a name="risk"/>

## 4. Риски

К основным рискам можно отнести:

*   **Зависимость от API:** Приложение полностью зависит от доступности и корректной работы бэкенд-сервера по адресу `http://localhost:8080/api/v2`. Если сервер не запущен или работает с ошибками, приложение будет неработоспособно.
*   **Целостность данных:** Удаление автора или категории может повлиять на связанные книги. Необходимо проверить, как система обрабатывает такие сценарии (например, не позволяет удалить автора, у которого есть книги).
*   **Совместимость с браузерами:** Тестирование будет проводиться в одном браузере (Google Chrome). Возможны проблемы с отображением или функциональностью в других браузерах.

<a name="features"/>

## 5. Аспекты тестирования

В ходе тестирования планируется проверить реализацию всех основных функций приложения.

### Общие аспекты

*   **Запуск приложения:** Проверка корректной загрузки приложения и отображения начального экрана (вкладка "Books").
*   **Загрузка данных:** Проверка, что при запуске и обновлении страницы корректно загружаются списки книг, авторов и категорий.
*   **Навигация:** Проверка переключения между вкладками "Books", "Authors", "Categories".

### Управление Книгами (Books)

*   **Отображение списка:** Проверка корректного отображения таблицы с книгами, включая название, автора и категории.
*   **Добавление книги:**
    *   Позитивный сценарий: успешное добавление книги с валидными данными.
    *   Негативный сценарий: проверка валидации (попытка сохранить без названия, без автора, с превышением длины названия).
*   **Редактирование книги:** Проверка открытия модального окна с предзаполненными данными и успешного сохранения изменений.
*   **Удаление книги:** Проверка появления диалога подтверждения и успешного удаления книги после подтверждения.
*   **Поиск:** Проверка фильтрации списка книг по названию и автору.

### Управление Авторами (Authors)

*   **Отображение списка:** Проверка корректного отображения таблицы с авторами и списком их книг.
*   **Добавление автора:**
    *   Позитивный сценарий: успешное добавление автора.
    *   Негативный сценарий: проверка валидации (пустое имя, превышение длины).
*   **Редактирование автора:** Проверка открытия модального окна с именем автора и успешного сохранения изменений.
*   **Удаление автора:** Проверка появления диалога подтверждения и успешного удаления.
*   **Поиск:** Проверка фильтрации списка авторов по имени.

### Управление Категориями (Categories)

*   **Отображение списка:** Проверка корректного отображения таблицы с категориями и списком их книг.
*   **Добавление категории:**
    *   Позитивный сценарий: успешное добавление категории.
    *   Негативный сценарий: проверка валидации (пустое имя, превышение длины).
*   **Редактирование категории:** Проверка открытия модального окна с данными категории и успешного сохранения изменений.
*   **Удаление категории:** Проверка появления диалога подтверждения и успешного удаления.
*   **Поиск:** Проверка фильтрации списка категорий по названию.

### Нефункциональные требования

*   **Индикаторы загрузки:** Проверка отображения спиннеров во время загрузки данных в таблицах и выпадающих списках.
*   **Сообщения об ошибках:** Проверка отображения всплывающих сообщений при ошибках API (например, при попытке создать сущность с уже существующим именем).
*   **Сообщения об успехе:** Проверка отображения всплывающих сообщений после успешного выполнения операций (создание, обновление, удаление).
